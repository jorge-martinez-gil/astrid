# ASTRID Tabular Fault-Injection Experiment Dossier

Date: 2026-05-19  
Experiment: controlled tabular reliability fault injection  
Tool state: ASTRID headless tabular analyzer in `astrid_core.py`  
Primary output directory: `experiments/results/tabular_fault_injection/`

## Purpose

This dossier records the first reproducible empirical experiment for a future
top-tier ASTRID paper. The experiment evaluates whether ASTRID detects known,
controlled reliability failures in tabular datasets and whether its health
score, verdict, policy gate, and recommendations respond in the expected
direction as fault severity increases.

This is seed evidence for the paper claim:

> A modality-aware pre-training dataset audit can surface measurable reliability
> risks before they propagate into model development workflows.

This experiment supports RQ1 from `docs/paper_blueprint.md`:

> How accurately does ASTRID detect injected dataset reliability failures across
> modalities?

It also gives early evidence for recommendation coverage and score monotonicity,
but it is not yet sufficient alone for a top-tier paper.

## Synthetic Dataset Protocol

The clean dataset is generated by `make_clean_dataset()` in
`experiments/fault_injection_tabular.py`.

Dataset size:

- Requested clean rows: 4,000.
- Rows may increase for duplicate and split-leakage injections because those
  faults append corrupted copies.

Columns:

- `sample_id`: synthetic identifier.
- `timestamp`: deterministic temporal slices across four months.
- `split`: train, validation, or test.
- `site`: synthetic industrial plant group.
- `sensor_temp`, `sensor_pressure`, `vibration_rms`, `humidity`: numeric sensor
  features.
- `operator_note`: text field for PII injection.
- `target`: binary label.

ASTRID configuration:

- Analyzer: tabular.
- Preset: `Balanced (recommended)`.
- Mode: `Quick Scan`.
- Label column: `target`.
- Split column: `split`.
- Time column: `timestamp`.
- Group column: `site`.
- ID column: `sample_id`.

## Fault-Injection Protocol

Each fault is injected independently into the clean dataset. This isolates the
effect of one failure mode at a time.

| Fault Type | Injected Failure | Primary ASTRID Metric |
| --- | --- | --- |
| `missingness` | Random missing numeric cells. | `overall_missing_rate` |
| `duplicates` | Exact duplicate row insertion. | `exact_duplicate_row_rate` |
| `split_leakage` | Duplicated samples moved across train/test splits. | `row_hash_cross_split_rate` |
| `drift` | Numeric feature shifts in the final temporal slice. | `max_numeric_drift_ks` |
| `pii` | Synthetic email-like strings in `operator_note`. | `pii_flagged_columns` |
| `fairness` | Group-dependent positive-rate disparity. | `positive_rate_disparity` |

Severity grid:

| Severity Values |
| --- |
| `0`, `0.01`, `0.05`, `0.10`, `0.20`, `0.40` |

Detection thresholds used by the summary script:

| Fault Type | Detection Rule |
| --- | --- |
| `missingness` | `overall_missing_rate > 0.05` |
| `duplicates` | `exact_duplicate_row_rate > 0.01` |
| `split_leakage` | `row_hash_cross_split_rate > 0` |
| `drift` | `max_numeric_drift_ks > drift_ks_threshold` |
| `pii` | `pii_flagged_columns > 0` |
| `fairness` | `positive_rate_disparity > 0.20` |

## Runtime

Total runtime over 36 runs: 7.733 seconds  
Mean runtime per run: 0.215 seconds  
Maximum runtime per run: 0.428 seconds

This suggests the tabular analyzer is fast enough for CI-style dataset gates on
small-to-medium tabular datasets. Larger scaling experiments are still required.

Interpretation:

- The score decreases under every injected fault family.
- PII and split leakage cause sharp early penalties, reflecting their critical

- Fairness lowers the score but remains relatively high; the scoring model may
  need stronger fairness penalties for governance-centered venues.

## Paper-Ready Claims Supported by This Run

Conservative claims supported:

1. ASTRID detects injected tabular PII and split leakage at low severity without
   false positives on the clean controls.
2. ASTRID's primary metrics are monotonic with injected severity for all six
   controlled tabular fault families.
3. ASTRID's global health score decreases under all injected reliability
   failures.
4. ASTRID recommendations are triggered when the corresponding fault crosses the
   active detection threshold.
5. Runtime is low enough for initial CI/MLOps use on this synthetic tabular
   benchmark.

Claims not yet supported:

1. ASTRID outperforms baseline tools.
2. ASTRID generalizes to real industrial datasets.
3. ASTRID score degradation correlates with downstream model degradation.
4. ASTRID's thresholds are empirically optimal.
5. ASTRID detects time-series and image faults with the same behavior.

## Threats to Validity

Internal validity:

- The synthetic dataset is intentionally controlled, so results may be easier
  than messy real industrial data.
- Faults are injected independently; real datasets often contain interacting
  failures.
- Detection labels use ASTRID-aligned thresholds. A baseline-independent ground
  truth scoring protocol should be added.

External validity:

- This run covers only tabular data.
- The dataset is synthetic and does not capture domain-specific units, sensors,
  seasonality, or operational constraints.
- PII is represented by email-like strings only.

Construct validity:

- ASTRID's health score is a decision-support index, not a proof of dataset
  safety.
- Fairness is represented by group positive-rate disparity. Other fairness
  concepts are not tested here.
- PII severity is summarized as flagged-column count, which saturates early.

Conclusion validity:

- No confidence intervals are reported yet.
- No repeated seeds are included yet.
- No baseline comparison is included yet.

Strongest framing:
